## 📌 비즈니스 요구사항

- 회원 가입 / 조회 가능
- 회원 등급: 일반, VIP
- 회원은 상품 주문 가능

### 할인 정책
- VIP는 1000원 할인 (고정 금액)
- 할인 정책은 변경 가능성이 높음


## 🤔 설계 고민

- 회원 데이터 저장 방식 미정 (DB / 외부 시스템)
- 할인 정책이 자주 바뀔 가능성 있음

👉 따라서 구현을 미루지 않고,
👉 변경에 유연한 구조 필요

## 💡 해결 전략

- 인터페이스 기반 설계
- 구현체를 언제든 교체 가능하도록 설계

예시:
- MemberRepository ← 인터페이스
- MemoryMemberRepository ← 구현체

- DiscountPolicy ← 인터페이스
- FixDiscountPolicy ← 구현체
